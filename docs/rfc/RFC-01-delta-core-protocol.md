# RFC-01: DELTA Core Protocol

**Status:** Draft  
**Version:** v2.5.0  
**Scope:** Core proof model, record structure, verification model, and security boundaries  
**Project:** DELTA Protocol  
**Tag target:** `v2.5.0-rfc-01-core-protocol`

## 1. Abstract

DELTA Protocol is an open cryptographic Proof of Change protocol.

DELTA is designed to prove that a specific change record existed, was structured in a verifiable way, was bound to cryptographic hashes, and could be independently checked according to declared verification methods.

The core model is:

```text
Before → Action → After → Evidence → Verification → Ledger
```

DELTA does not attempt to be a cryptocurrency, token, SaaS platform, marketplace, account system, or blockchain. DELTA is a protocol and reference implementation for producing and verifying cryptographic records of change.

## 2. Motivation

Modern systems can often prove ownership, authorship, commit history, or execution logs. They are weaker at proving a complete change event:

- what existed before;
- what action was taken;
- what existed after;
- what evidence supports the change;
- how verification was performed;
- who or what attested intent, audit, publication, trust, or wallet control;
- whether the proof can be replayed or independently checked.

DELTA aims to make change itself a first-class cryptographic object.

## 3. Design Goals

DELTA is designed to be:

1. **Zero-token:** no token, coin, gas, or economic dependency is required.
2. **Open:** implementable by independent tools and organizations.
3. **Hash-first:** all critical proof objects are bound by cryptographic hashes.
4. **Signature-aware:** records can be signed and verified using public/private keys.
5. **Evidence-preserving:** private evidence can remain private while still being committed by hash.
6. **Replay-capable:** declared verification methods can be rerun where practical.
7. **Audit-friendly:** evidence can be encrypted for auditors.
8. **Publication-aware:** records can be anchored or published without changing the core proof.
9. **Trust-chain capable:** multiple trust events can be chained.
10. **Wallet-aware:** address-control proofs can be bound to records.

## 4. Non-Goals

DELTA does not, by itself, prove:

- legal truth;
- real-world truth;
- identity of a natural person;
- legal ownership of a wallet or asset;
- regulatory compliance;
- correctness of external data sources;
- authority of an auditor, verifier, regulator, or signer;
- that a change was morally, legally, or commercially acceptable;
- that a record was generated by an uncompromised machine;
- that private evidence is true, only that its hash or encrypted form is bound to the record.

These properties may be supported by external legal, organizational, or technical processes, but they are not guaranteed by the DELTA core protocol alone.

## 5. Terminology

### 5.1 DELTA Record

A canonical proof object describing a change and its verification metadata.

### 5.2 Record Body

The signed or hash-bound core content of a record.

### 5.3 Record Hash

The SHA-256 hash of the full canonical `delta-record.json`, unless a narrower field is explicitly specified by a profile.

### 5.4 Evidence

Files, logs, commands, outputs, manifests, private payloads, or external proof objects supporting the claimed change.

### 5.5 Measurement Method

A declared method used to measure or verify some part of the change. Examples include CLI verification, file audit, replay, security scan, wallet proof, or benchmark.

### 5.6 Replay

A process that attempts to reproduce verification in an isolated or fresh environment.

### 5.7 Intent

A detached or embedded proof that a human or authorized system intended or approved a specific change.

### 5.8 Audit Package

An encrypted evidence package intended for a named auditor public key.

### 5.9 Publication Proof

A proof that a record hash was published, timestamped, anchored, or externally referenced.

### 5.10 Trust Ledger

A hash-chain ledger of trust events, such as execution, verification, audit, approval, or regulatory review.

### 5.11 Wallet Proof

A proof that a cryptographic address or key signed a DELTA challenge bound to a specific record hash.

## 6. Canonicalization

DELTA proof objects should be serialized using canonical JSON.

A conforming implementation should:

- use UTF-8;
- sort object keys consistently;
- avoid non-deterministic whitespace;
- avoid platform-dependent line endings in hashed JSON;
- hash exact canonical bytes, not pretty-printed display output.

The default hash algorithm is:

```text
sha256:<64 lowercase hex characters>
```

## 7. Core Record Model

A DELTA record should include:

```json
{
  "type": "delta_record",
  "version": "1.0.0",
  "record_body": {},
  "record_body_hash": "sha256:...",
  "signature": {},
  "verification": {},
  "evidence": {}
}
```

The exact schema may vary by profile, but the core security requirement is that critical claims are hash-bound and verifiable.

## 8. Core Hash Binding Rule

For modern DELTA proof modules, external proofs should bind to the SHA-256 hash of the full canonical `delta-record.json`.

This rule is used by:

- Proof of Intent;
- Proof of Audit;
- Proof of Publication;
- Proof of Trust;
- Proof of Wallet.

Older or narrower modules may reference `record_body_hash`, but full-record binding is preferred for new profiles because it reduces ambiguity and prevents selective rebinding.

## 9. Signature Model

DELTA supports signed records and signed auxiliary proof objects.

A signature object should identify:

- signing algorithm;
- public key or public key reference;
- public key hash;
- signature;
- signed payload hash;
- verification status.

A verifier should fail closed when:

- the signature is malformed;
- the public key does not match its hash;
- the signed payload hash does not match the payload;
- the signature does not verify;
- the record or proof body self-check fails.

## 10. Evidence Model

Evidence may be:

- public evidence;
- private evidence commitments;
- encrypted auditor evidence;
- external evidence;
- generated artifacts;
- logs;
- manifests;
- commands;
- replay scripts;
- external publication or wallet proof objects.

Evidence does not need to be public to be useful. A private evidence hash can prove that a specific private evidence item existed and was bound to a record, without revealing the evidence.

## 11. Verification Model

A DELTA verifier should report structured checks, not only a single pass/fail result.

Examples:

```text
DELTA_VERIFY_OK=True
DELTA_RECORD_SIGNATURE_OK=True
DELTA_RECORD_BODY_HASH_OK=True
DELTA_EVIDENCE_HASHES_OK=True
DELTA_REPLAY_RESULT=OK
DELTA_RECORD_BINDING_OK=True
```

A verifier should also report reasons for failure:

```text
DELTA_REASON_RECORD_BINDING_OK=record_hash_mismatch
DELTA_REASON_SIGNATURE_OK=signature_invalid
```

This makes DELTA useful for automated systems and human auditors.

## 12. Proof of Replay

Proof of Replay attempts to re-run a declared verification method.

A replay process may:

- clone a repository into a fresh directory;
- check out the declared commit;
- install dependencies or explicitly skip installation;
- execute replay instructions;
- verify stdout/stderr hashes;
- verify measurement result;
- verify after-state hash;
- verify method definition hash;
- report success/failure.

Replay does not prove that the original environment was honest. It proves whether an independent replay procedure reproduces the declared result under the verifier's conditions.

## 13. Proof of Intent

Proof of Intent binds a conscious approval or authorization object to a specific DELTA record.

An intent proof should include:

- intent body;
- target full record hash;
- signer public key;
- signature;
- registry entry or key reference;
- policy metadata where applicable.

Proof of Intent proves that a specific key signed a specific intent object bound to a specific DELTA record hash. It does not prove legal authority unless an external governance system gives that key legal meaning.

## 14. Intent Policy

A DELTA record may declare intent policy:

```json
{
  "intent_required": true,
  "intent_deadline": "2026-05-19T00:00:00Z",
  "policy_id": "intent-policy-v1",
  "enforcement": "report_only"
}
```

Current policy reporting may be non-enforcing. Enforcing profiles may later require that missing or invalid intent causes verification failure.

## 15. Proof of Audit

Proof of Audit encrypts private evidence for an auditor while keeping public verification possible.

A valid audit package should bind to:

- the full DELTA record hash;
- auditor public key or key hash;
- encrypted evidence entries;
- ciphertext hashes;
- AAD hashes;
- package self-checks.

Proof of Audit proves that encrypted evidence was packaged for a specific auditor key and bound to a specific DELTA record. It does not prove that the auditor is legally authorized or that decrypted evidence is true.

## 16. Proof of Publication

Proof of Publication binds a DELTA record hash to a publication proof object.

Publication methods may include:

- local timestamp proof;
- GitHub release asset proof;
- OpenTimestamps pending proof;
- external anchoring methods;
- future mirror nodes.

Proof of Publication proves that a proof object refers to a specific record hash and, depending on backend, may support an existence-before-time claim. The strength of the time claim depends on the publication method.

## 17. Proof of Trust

Proof of Trust is a hash-chain ledger of trust events.

Each entry should include:

- entry id;
- actor;
- role;
- event type;
- linked record hash;
- previous entry hash;
- entry body hash;
- self-check.

A trust ledger should include:

- chain id;
- ledger body;
- ledger body hash;
- last entry hash;
- entries.

Proof of Trust proves hash-chain consistency and record binding. It does not prove that an actor truly had real-world authority.

## 18. Proof of Wallet

Proof of Wallet binds a wallet/address-control proof to a DELTA record.

Supported profiles may include:

- `ed25519_address_control_v1`;
- `ethereum_eip191_personal_sign_v1`;
- `ethereum_eip712_typed_data_v1`;
- `bitcoin_bip322_external_v1`.

Proof of Wallet proves that a key/address signed or supplied a proof object bound to a specific record hash, subject to the verification level of that wallet standard.

For Bitcoin `bitcoin_bip322_external_v1`, local cryptographic signature verification is explicitly not claimed. Verification is shape-only/external-pending until a local BIP-322 verifier profile is implemented.

## 19. Security Boundary Summary

DELTA proves cryptographic relationships between hashes, signatures, evidence, records, and proof objects.

DELTA does not replace:

- legal contracts;
- identity proofing;
- compliance certification;
- key custody procedures;
- secure hardware;
- trusted timestamp authorities;
- auditors;
- regulators;
- organizational governance.

DELTA should be used as a cryptographic evidence layer within broader systems.

## 20. Conformance Levels

### Level 0: Hash Record

Implementation can canonicalize and hash DELTA records.

### Level 1: Signed Record

Implementation can verify record signatures and self-checks.

### Level 2: Evidence Binding

Implementation can verify evidence hashes and method definitions.

### Level 3: Replay

Implementation can run declared replay workflows.

### Level 4: Intent / Audit / Publication

Implementation can verify auxiliary proof bundles bound to the full record hash.

### Level 5: Trust / Wallet

Implementation can verify trust ledgers and wallet/address-control proofs.

### Level 6: Policy Enforcement

Implementation can enforce organization-specific or standard-specific policies.

## 21. Versioning

DELTA proof modules should use explicit versions.

Examples:

```text
delta_record_v1
intent_attestation_i001
proof_of_audit_v1
publication_proof_v1
delta_trust_ledger_v1
ethereum_eip712_typed_data_v1
bitcoin_bip322_external_v1
```

Breaking schema changes should increment the relevant proof/module version.

## 22. Reference Implementation

The current reference implementation is a Python CLI-oriented technical alpha.

It is suitable for:

- demonstrations;
- test vectors;
- protocol exploration;
- security review;
- early integrations;
- documentation.

It is not yet a production enterprise system.

## 23. Future Work

Planned work includes:

## 24. Summary

DELTA is a protocol for proving change, not merely storing logs.

It turns a change event into a cryptographic object that can be hashed, signed, replayed, audited, published, chained, and bound to wallet/address-control proofs.

The central claim is:

```text
The internet can prove ownership.
DELTA proves change.
```
