# DELTA v2.16.7 — GitHub Actions Demo Workflow / Automated Tamper Detection Check

## Executive summary

DELTA v2.16.7 introduces the first GitHub Actions demonstration workflow for DELTA Protocol.

After v2.16.6 added a minimal runnable public tamper-detection artifact, this milestone turns that local demonstration into an automated CI check that can be observed through GitHub Actions.

The purpose of this milestone is to move DELTA one step further from documentation-only public readiness toward observable, repeatable, automation-friendly public demonstration.

This release does not change cryptographic verification semantics, proof profiles, schemas, signatures, canonicalization rules, bundle rules, hash-binding rules, or implementation logic.

## Added files

- `.github/workflows/delta-minimal-demo.yml`
- `docs/integrations/github-actions-demo-workflow-v2.16.7.md`
- `docs/changelog/v2.16.7-github-actions-demo-workflow.md`

## What the workflow does

The workflow performs a focused set of public demonstration checks:

- runs the minimal public tamper-detection demo from `demo/minimal-demo/verify.ps1`;
- runs the Python reference baseline verification with `python src/delta_cli.py verify-all`;
- runs Canonical JSON / JCS vector verification with `python tools/delta_canonical_json.py verify-vectors --vectors tests/vectors/canonical-json/vectors.json`.

## Strategic purpose

DELTA’s public adoption phase requires visible verification.

A local demo is useful, but a CI-visible demonstration is stronger: it shows that DELTA’s basic tamper-detection walkthrough and core verification commands can run in a standard automated environment.

This milestone helps external reviewers, auditors, developers, and early adopters see DELTA as something that can enter real engineering workflows rather than remain only a protocol document.

## Scope

This milestone is intentionally narrow.

It does not implement a reusable marketplace GitHub Action.

It does not implement production signed bundle verification in CI.

It does not add new cryptographic proof semantics.

It does not claim legal identity, signer authority, regulatory compliance, real-world truth, sensor honesty, evidence completeness, policy correctness, institutional approval, or business authority.

It automates the minimal public tamper-detection demo and baseline verification commands.

## Security impact

Workflow and documentation only.

No cryptographic verification semantics are changed.

No proof profile is modified.

No verifier behavior is changed.

No implementation behavior is changed.

No new cryptographic functionality is added.

This milestone strengthens public reviewability by making the minimal demonstration observable in CI while preserving DELTA’s strict anti-overclaiming boundary.

## Expected output

A successful workflow run should include:

```text
[OK] Original artifact hash matches expected value.
[FAIL] Tampered artifact hash mismatch detected.
[OK] Demo succeeded: tampering was detected.
DELTA CLI RESULT: OK
DELTA_JCS_VERIFY_OK=True
```
