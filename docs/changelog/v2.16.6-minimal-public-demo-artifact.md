# DELTA v2.16.6 — Minimal Public Demo Artifact / Tamper Detection Walkthrough

## Executive summary

DELTA v2.16.6 introduces a minimal public demonstration artifact for DELTA Protocol.

After v2.16.1 established the Public Adoption Freeze, v2.16.2 introduced the public landing and quick-start layer, v2.16.3 documented CI/CD and public demonstration pathways, v2.16.4 added a Public Reviewer Checklist, and v2.16.5 introduced the External Review Request / Public Feedback Intake path, this milestone adds a small, runnable tamper-detection walkthrough.

The purpose of this release is to allow a first-time reviewer to personally observe the core tamper-evidence intuition behind DELTA in approximately two minutes:

```text
known artifact hash -> verification OK -> modify artifact -> hash mismatch -> verification FAIL
```

This release is intentionally conservative.

It does not introduce a mock signed `.delta` bundle and does not claim to provide full signed bundle verification. The demo is a public onboarding artifact that demonstrates the hash-binding principle underlying DELTA’s Proof of Change model.

## Strategic purpose

DELTA’s public adoption phase now requires a simple demonstration that can be shown before a reviewer reads deeper RFC, schema, verifier, or security-boundary documentation.

This milestone provides that demonstration while preserving DELTA’s anti-overclaiming discipline.

The demo is intentionally minimal because its job is not to replace the reference implementation. Its job is to make the first concept visible: cryptographic binding makes later tampering detectable.

## Demo scope

The demo shows:

- verification of a known artifact hash,
- creation of a temporary working copy,
- deliberate tampering with that working copy,
- detection of the resulting SHA-256 mismatch,
- a clear public explanation of what the demo does and does not prove.

The demo also points users toward the full DELTA baseline verifier:

```text
python src/delta_cli.py verify-all
```

## Security boundary

This release does not add new cryptographic proof functionality.

It does not change DELTA verification semantics.

It does not modify proof profiles, schemas, signatures, canonicalization rules, bundle rules, hash-binding rules, or verifier behavior.

The demo is not:

- a signed `.delta` bundle verifier,
- a legal proof,
- a compliance proof,
- a proof of signer authority,
- a proof of real-world truth,
- a proof of sensor honesty,
- a proof of evidence completeness,
- a substitute for DELTA canonical JSON validation,
- a substitute for Ed25519 signature verification,
- a substitute for full DELTA bundle verification.

## Why no mock signed bundle

This milestone intentionally avoids publishing a fake signed bundle with mock signatures.

Using mock signatures in a public proof demo could weaken DELTA’s credibility by making the demonstration appear stronger than it is.

Instead, v2.16.6 uses an honest minimal artifact with an explicit boundary: it demonstrates tamper detection at the hash layer and directs reviewers to the full DELTA verifier stack for deeper verification.

DELTA continues its public adoption and external-review phase.

The project now has:

- a formal public adoption freeze,
- a public landing and quick-start layer,
